# InfiniteMail 公司邮箱闭环

本文档记录 InfiniteMail 从前端工作台推进到公司自用邮箱系统的落地边界。当前代码已进入 BFF 真实接入闭环：管理后台配置邮箱域名、邮箱名前缀、OAuth 展示名称、邮箱/手机号登录注册、注册码开关、阿里云短信参数和邮件底座；用户端支持 OAuth、手机号验证码加密码、邮箱密码登录注册，以及注册码注册指定邮箱。

## 开源底座选择

首选底座：Stalwart Mail Server。

- 适合原因：单体覆盖 SMTP、IMAP、JMAP、管理能力和现代邮件服务器能力，适合我们自建域名邮箱并继续做平台化管理。
- 产品定位：它做真实邮件收发、投递、邮箱协议、存储、认证和域名能力；InfiniteMail 做公司账号、注册策略、管理后台、用户邮箱 UI 和业务邮箱能力。
- 官方资料：https://stalw.art/docs/

第二参考：WildDuck。

- 适合借鉴：REST 管理 API、账号/邮箱管理、Node.js 生态和多租户邮箱产品的接口形态。
- 不作为第一选择的原因：我们更需要一个可直接承担完整邮件服务器职责的现代底座；WildDuck 更适合参考管理 API 和应用层模型。
- 官方仓库：https://github.com/nodemailer/wildduck

部署型参考：mailcow。

- 适合借鉴：生产邮件套件、域名、DKIM、反垃圾和运维面板体验。
- 不作为第一选择的原因：它更像完整运维套件，二次开发成我们平台内置产品时边界更重。
- 官方资料：https://docs.mailcow.email/

## 当前已实现

- 管理后台入口：`http://localhost:1888`
- 用户端入口：`http://localhost:1788`
- API/BFF 入口：`http://localhost:1666`
- 邮箱域名配置：后台保存 `mailbox.domain`
- 强制前缀策略：开启后只允许 `user-name@domain`、`admin-name@domain` 等；关闭后允许 `name@domain`
- 前缀列表配置：后台保存 `mailbox.allowedPrefixes`
- OAuth 名称配置：后台保存 `auth.oauthProviderName`，用户端展示为“使用 xxx 账号登录”
- 普通账号密码登录：手机号 + 验证码 + 密码，或公司邮箱 + 密码
- 普通注册：手机号 + 验证码 + 密码 + 邮箱名
- 注册码注册：后台生成指定邮箱注册链接，用户带注册码注册
- 短信验证码记录：后台可看到手机号、用途、发送状态、生成时间；本地调试展示验证码，阿里云/生产模式只展示脱敏验证码，服务端仅用哈希校验
- 阿里云短信真实发送：AccessKey、签名、模板 Code、有效期和启用开关均由后台保存，开启后 BFF 直接调用阿里云短信接口
- Go BFF 已支持 `DATABASE_URL`：自动执行 PostgreSQL migrations，并把配置、账号、注册码、短信日志、会话、设置和基础邮件写入规范化业务表
- 新注册密码已使用 Argon2id 强哈希；旧 salted SHA256 哈希保留验证兼容，便于历史账号平滑迁移
- Session 持久化使用 token hash 作为查询键，后台支持管理员密码登录、短期管理会话和 `X-Admin-Token` 自动化令牌，CORS 预检已放行后台令牌请求
- 管理后台已增加独立“审计日志”导航，记录后台配置、邀请、验证码、注册、登录、邮箱激活和设置变更
- 管理后台账号记录已支持禁用/启用账号、重置临时密码；禁用和重置会清理账号现有会话，登录会拦截被禁用账号
- 管理后台域名邮箱页已支持 MX/SPF/DKIM/DMARC 建议记录和 DNS 验证状态，BFF 会查询公网 DNS 并保存检查结果
- 管理后台域名邮箱页已支持邮件服务连接配置、开通接口路径、账号生命周期接口、管理令牌脱敏保存与连接测试；账号记录展示邮箱开通状态，并可重试开通
- 管理后台任务中心已支持邮箱开通队列处理；注册、OAuth 首次开通和手动重试都会进入可追踪任务
- 任务中心可在后台保存自动执行开关和间隔，BFF worker 会按配置自动巡检并保存最近执行时间、状态和说明
- 邮箱开通队列已支持真实 HTTP provisioning：BFF 向配置的开通接口 `POST` 账号 payload，带 `Idempotency-Key` 和管理令牌；`2xx` 成功，`409` 按账号已存在处理
- 账号禁用、启用、重置密码已支持真实 HTTP lifecycle 同步：BFF 向配置的生命周期接口发送 `disable`、`enable`、`reset_password` action，远端失败时本地状态不会提前落库
- Stalwart 直连开通已接入：选择 Stalwart、填写服务地址和管理员 Token 后，即使不配置自定义开通/生命周期接口，BFF 也会通过 Stalwart JMAP 管理接口创建域名、创建邮箱账号、禁用/启用认证和重置邮箱密码
- 用户端邮件数据面已接入 BFF：收件列表、详情、`/messages/send`、`/drafts`、附件、星标、移动、已读状态和投递追踪接口会落本地快照，用户端收件箱、已发送、草稿箱、未读数和详情可直接读取
- 用户端账号安全已接入 BFF：设置页会拉取真实登录设备记录，展示设备、IP、最近使用时间，并支持移除单个设备或退出其他设备；PostgreSQL 会保存会话设备字段，token 仍只以 hash 形式落库
- 邮件服务数据面接口已预留：后台可配置收件列表接口、邮件详情接口、发信接口、草稿接口、星标接口、移动接口、已读接口；填完整 URL 或相对路径均可，BFF 会携带账号、邮箱、收件人、正文、附件元数据、附件 base64 兜底内容、读状态、`Idempotency-Key` 和管理令牌转发给真实邮件底座，并保存 `providerMessageId`、`acceptedAt` 和投递错误字段用于后续回调对账
- SMTP 直连发信已接入：后台可配置自建邮件服务器的 25/587/465、TLS 模式和 SMTP 账号密码；没有 HTTP 发信接口时，BFF 仍可生成标准 MIME 邮件并通过 SMTP 投递 To/Cc/Bcc
- IMAP 直连收件已接入：后台可配置自建邮件服务器的 143/993、TLS 模式、账号模板和密码；没有 HTTP 收件数据面时，BFF 会通过 IMAP 拉取收件列表/详情、同步星标/已读/移动，并把草稿追加到 Drafts 文件夹
- 自动回复真实投递已接入：用户在设置页开启自动回复后，收信 Webhook 写入收件箱时会通过已配置的 HTTP 发信接口或 SMTP 通道自动回复原发件人；重复 Webhook 不会重复发送同一封自动回复

## 生产落地顺序

1. 先部署 Stalwart 作为真实邮件底座，完成域名、MX、SPF、DKIM、DMARC、TLS。
2. 新建 InfiniteMail 后端，保存后台配置、账号、注册码、短信日志和邮箱映射。
3. 后端对接 Stalwart 管理能力，创建域名、邮箱账号和密码/令牌。
4. 用户注册成功后，先落平台账号，再写入邮箱开通任务；任务中心或后台 worker 负责创建邮箱账号并回写状态，最后返回邮箱工作台 session。
5. 发信/收信优先走 JMAP 或后端 API 聚合；已有自建 SMTP/IMAP 时可以先用 BFF 直连 SMTP 发信、IMAP 收信和草稿/读状态同步，再逐步补齐更完整的 JMAP/HTTP 数据面。
6. 阿里云短信走后端发送，验证码只保存哈希、用途、手机号、过期时间和校验结果。
7. 邮箱开通队列使用 `provisionJobs` 记录账号、邮箱、状态、尝试次数、错误和完成时间，避免注册链路被邮件底座短暂故障拖垮。
8. 自动巡检配置保存在 `adminConfig.ops`，包含 `autoRunEnabled`、`intervalMinutes`、`lastRunAt`、`lastRunStatus` 和 `lastRunMessage`。
9. 真实邮件底座接入时，管理后台配置 `mailbox.server.baseUrl`、`mailbox.server.provisionPath`、`mailbox.server.lifecyclePath`、`mailbox.server.messageListPath`、`mailbox.server.messageDetailPath`、`mailbox.server.messageSendPath`、`mailbox.server.draftPath`、`mailbox.server.messageStarPath`、`mailbox.server.messageMovePath`、`mailbox.server.messageReadPath` 和管理令牌；BFF 负责脱敏返回令牌状态，不能把令牌回传给前端。
10. 开通接口必须按 `provisionJobId` 或 `Idempotency-Key` 做幂等；重复创建返回 `409` 时 BFF 会按成功处理并写入外部账号 ID。
11. 生命周期接口必须识别 `action` 字段：`disable` 暂停邮箱账号，`enable` 恢复邮箱账号，`reset_password` 使用请求体内的临时密码更新邮箱账号密码。
12. 收件列表、详情、发信、草稿、星标、移动和已读接口必须识别账号上下文；写操作还必须识别 `Idempotency-Key`。若启用 SMTP+IMAP，协议配置可替代对应 HTTP 数据面。开启生产严格模式后，缺失的数据面不会返回兼容结果。
13. 收信 Webhook 可携带 `headers` 字段；BFF 会识别 `Auto-Submitted`、`Precedence`、`List-Id`、`X-Auto-Response-Suppress` 等头，避免对系统邮件、列表邮件和自动回复邮件再次自动回复。

## 安全要求

- 生产环境禁止在前端保存 AccessKey Secret、邮箱密码和验证码明文。
- 验证码后台只展示调试环境明文；生产只展示脱敏和发送状态。
- 密码必须使用服务端强哈希，例如 Argon2id 或 bcrypt。
- 注册码必须有过期时间、使用状态、绑定手机号和审计记录。
- 管理后台必须独立鉴权，并支持管理员操作审计。
- 用户登录会话必须支持设备留痕、最近使用时间、IP 记录和主动踢出其他设备。
- 邮箱域名切换必须校验 DNS 配置，避免用户注册到未验证域名。
- 邮箱开通必须可重试、可审计、可追踪；任务失败不能丢失账号，也不能反复无状态重试。
